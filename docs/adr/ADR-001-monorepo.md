# ADR-001: Monorepo 架构

## 状态

Accepted

## 背景

团队规模 1-3 人，需要清晰的项目结构，但不想引入微服务的复杂度。

## 决策

采用 Monorepo + 模块化单体架构：

- **Monorepo**：使用 pnpm workspace + Turborepo 管理
- **apps/web**：Next.js 主站
- **apps/api**：FastAPI 核心服务
- **apps/worker**：Celery 异步任务
- **packages/\***：共享包

## 理由

1. 适合小团队：结构清晰，不引入过度复杂度
2. 易于协调：代码变更在一个仓库内可见
3. 渐进式拆分：后续可按热点模块逐步拆服务
4. 共享包管理：类型契约、UI 组件等共享方便

## 影响

- 正面：简化依赖管理，减少跨仓库协调
- 负面：CI 时间可能较长（可通过 Turborepo 缓存优化）

## 替代方案

### 多仓库

- 优点：独立部署，CI 更快
- 缺点：跨仓库协调困难，共享包管理复杂
- 结论：不适合小团队

### 单一代码仓库（无 workspace）

- 优点：最简单
- 缺点：前端后端混合，依赖冲突
- 结论：不适合双运行时场景
