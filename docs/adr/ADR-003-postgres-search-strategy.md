# ADR-003: PostgreSQL 混合检索策略

## 状态

Accepted

## 背景

项目需要搜索功能，但 Elasticsearch：
- 部署复杂度高
- 运维成本高
- 数据同步延迟

## 决策

v1 使用 PostgreSQL 原生能力实现搜索：

- **词法检索**：`tsvector` + GIN 索引
- **语义检索**：`pgvector` 扩展
- **混合排序**：加权合并两种检索结果

## 理由

1. **简单部署**：PostgreSQL 已是主数据库，无需额外部署
2. **数据一致**：搜索索引与业务数据在同一数据库，无同步延迟
3. **足够性能**：Postgres 全文检索 + 向量检索对中小规模数据足够
4. **渐进升级**：后续数据量增长可平滑迁移到 Elasticsearch

## 影响

- 正面：降低运维复杂度，保持数据一致性
- 负面：大规模数据可能性能不足

## 迁移条件

满足以下条件时考虑迁移到 Elasticsearch：
- 搜索文档数量 > 100 万
- 搜索响应时间 > 500ms
- 需要更复杂的聚合/分析

## 中文支持

需配置中文分词：
- 方案一：zhparser 扩展
- 方案二：jieba 分词 + 自定义词典
